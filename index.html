Trace-based vs function-extraction decompilation

Lutetia uses a trace-based approach, not function extraction. You may see fewer named functions than in tools like Ethervm; the logic is still there, just inlined.

Ethervm-style (function extraction): The decompiler finds internal jump targets (e.g. func_0899, func_08FE) and prints them as separate function func_XXXX(...) with memory[] / var / goto. The main code then looks like calls to those helpers, so you get many small, named functions.

Lutetia (trace-based): It follows execution from the entry point and builds one (or a few) linear traces. The same logic is inlined in that trace: those checks become require / revert with, lookups become conditionals (if not _69001: ... elif _69001 == 1: ...), and helper-style logic appears where it is used. You get one main def fallback(): (and possibly other entry points) with everything inlined.

Summary: Lutetia does not omit behavior; it does not extract it as separate functions. Ethervm shows “more functions” because it lists every such block as a function. Lutetia shows one (or few) large function(s) with the same behavior inlined. Trade-off: many small functions can clarify “which helper does what”; one inlined trace can make the main flow easier to follow without jumping between definitions.
